# RxnNet
An AI framework for automated reaction mechanism discovery.

This is the repository of the Rxnnet program package developed by the Major group.
The code uses two external packages, which must be installed locally on your machine

1) geodesic_interpolate (https://github.com/virtualzx-nad/geodesic-interpolate)
2) ORCA quantum chemistry package (https://orca-manual.mpi-muelheim.mpg.de/contents/quickstartguide/installation.html#installation)

Before cloning this repository, you need to install Git LFS (https://git-lfs.com/)

Clone the repository:
```text
git clone https://github.com/majordt/RxnNet.git
```

Add the following line to your `.bashrc` file:
```text
export MKL_THREADING_LAYER=GNU
```

# Configuration

All parameters for every mode are provided in a single YAML input file. By default, RxnNet reads `input.yaml` from the working directory. A custom file can be specified with the `--input` flag:

```text
./rxnnet --mode=run --input=my_input.yaml
```

Keys in the YAML file are **lowercase**. An annotated template is provided below and in the bundled `input.yaml`.

---

# Generating the reaction graph

Ensure that the required keywords are present in the YAML input file:

```yaml
molecule_name: rosalexin           # Name of the molecule
molecule_smile: 'C=C1CC...'        # SMILES string of the reactant
reaction_list_name: ring_closure   # Name of the reaction list variable in reactions.py
reactions_path: reactions.py       # Path to reactions.py (absolute or relative)
max_path_length_to_run: 2          # Depth of the reaction graph (number of steps)
```

Generate the reaction tree with:

```text
./rxnnet --mode=run --input=my_input.yaml
```

The reaction graph generated by RxnNet is saved as pickle files in `Results/MOLECULE_NAME`.

# RxnNet Analyzer
Extract the reaction mechanism from the graph and generate geodesic paths for each step of the mechanism.

Required keywords in the YAML input file:

```yaml
source_smile: '...'             # SMILES of the initial reactant
smile_to_analyze: '...'         # SMILES of the final product
                                # (mutually exclusive with mass_to_analyze)
# mass_to_analyze: 272.25       # Alternatively, target by molecular mass
# charge: 1                     # Required when using mass_to_analyze
max_path_length_to_analyze: 5   # Number of reaction steps between reactant and product
ts_num_frames: 18               # Number of intermediate geometries in the geodesic path
align_to_product: true          # true  → template-based approach
                                # false → distance-restraint-based approach
num_confs_to_generate: 100      # Number of conformers to generate per intermediate
```

Run the RxnNet analyzer with:

```text
./rxnnet --mode=analyze --input=my_input.yaml
```

TIP: If RxnNet fails to generate a path under the distance-restraint-based approach (`align_to_product: false`), try with the template-based approach (`align_to_product: true`).

RxnNet analyzer extracts the distinct paths connecting the source and target SMILES from the graph. Each path has a number of reaction steps equal to `max_path_length_to_analyze`. It then iterates through each path and generates geodesic paths after considering stereochemistry, regiochemistry, conformations, and atom labeling. Each RxnNet analyzer run creates a directory with the timestamp inside `Results/MOLECULE_NAME`. A directory tree for Trichodine is shown below:

```text
Results/
├── Trichodine
│   ├── 2026-03-18_22-04-02
│   │   ├── set0
│   │   ├── set1
│   │   ├── set2
```

Directories `set0`, `set1`, etc., contain the geodesic paths for the distinct paths generated by RxnNet. The geodesic paths generated for a five-step reaction are shown below (distance-restraint-based approach, `align_to_product: false`):

```text
set0/
├── 2D_path.png
├── conf_1p_to_2r.xyz
├── conf_2p_to_3r.xyz
├── conf_3p_to_4r.xyz
├── conf_4p_to_5r.xyz
├── conf_final_product.xyz
├── conf_initial_reactant.xyz
├── smiles.json
├── step_1.xyz
├── step_2.xyz
├── step_3.xyz
├── step_4.xyz
└── step_5.xyz
```

```text
step_1.xyz, step_2.xyz, etc   : geodesic paths for each chemical step in the set.
conf_1p_to_2r.xyz             : the product carbocation (CC) intermediate of step 1
                                and the reactant CC intermediate of step 2 are the
                                same system in different conformations. This file
                                contains the geodesic path for that conformational change.
conf_initial_reactant.xyz     : conformational change of the initial reactant from its
                                global minimum conformer to the reactant of step 1.
conf_final_product.xyz        : conformational change of the final product from the
                                product conformation of the last step to its global
                                minimum conformer.
```

# QM module: quantum chemistry calculations

Perform nudged elastic band (NEB) calculations with ORCA using the geodesic paths as the initial guess.

Required keywords in the YAML input file:

```yaml
folder_name: '2026-03-18_22-04-02'   # Timestamp directory inside Results/MOLECULE_NAME
qm_set_folders: set0                  # Set(s) to process: a string, a list, or omit
                                      # the key entirely to process all set* folders
# qm_set_folders: [set0, set1]
charge: 1                             # Charge of the system
multiplicity: 1                       # Multiplicity of the system
orca_path: '/path/to/orca'            # Path to the ORCA executable
setenv_orca:                          # Environment variables required by ORCA
  - 'LD_LIBRARY_PATH /path/to/openmpi/lib/:${LD_LIBRARY_PATH}'
  - 'PATH /path/to/openmpi/bin/:${PATH}'
  - 'PATH /path/to/orca/:${PATH}'
  - 'XTBEXE /path/to/xtb'            # xTB executable (for semi-empirical calculations)
```

Keywords for semi-empirical calculations:

```yaml
se: true                          # true/false – run semiempirical calculations at
                                  # GFNn-xTB level using the ORCA–xTB interface
semi_empirical_method: GFN2-xTB  # xTB method
se_initial_guess: true            # true/false – use geodesic paths as initial guess
                                  # for xTB NEB calculations
semi_empirical_keywords: []       # Additional xTB keywords (can be an empty list)
semi_empirical_nprocs: 12         # Number of processors for xTB calculations
cutoff: 100.0                     # Energy cutoff (kcal/mol) for filtering NEB images
```

If `se_initial_guess: true`, the xTB calculations are organized as:

```text
├── Trichodine
│   ├── 2026-03-18_22-04-02
│   │   ├── set0
│   │   │   ├── SE_QM_with_initial_guess
│   │   │   │   ├── step_1
│   │   │   │   │   ├── run_orca.sh
│   │   │   │   │   ├── set0_step_1_se_firstopt.inp
│   │   │   │   │   ├── set0_step_1_se_lastopt.inp
│   │   │   │   │   ├── set0_step_1_se_NEB.inp
│   │   │   │   ├── conf_1p_to_2r
│   │   │   │   │   ├── run_orca.sh
│   │   │   │   │   ├── set0_conf_1p_to_2r_se_firstopt.inp
│   │   │   │   │   ├── set0_conf_1p_to_2r_se_lastopt.inp
│   │   │   │   │   ├── set0_conf_1p_to_2r_se_NEB.inp
```

The reactant and product optimizations are carried out first (`_se_firstopt.inp`, `_se_lastopt.inp`), followed by the NEB calculation (`_se_NEB.inp`). xTB-level calculations are then optionally followed by DFT NEB.

Since DFT NEB calculations are expensive, it is also possible to generate only the DFT-level ORCA inputs with `dft_input_only: true`.

Keywords for DFT calculations:

```yaml
dft: false                        # true/false – run DFT-level NEB
dft_initial_guess: true           # true/false – use xTB NEB MEPs as initial guess
                                  # for DFT NEB
dft_input_only: true              # true/false – write DFT ORCA inputs only,
                                  # do not run ORCA
dft_method: M062X                 # DFT functional
dft_basis_set: def2-SVP           # Basis set
dft_keywords: [defgrid3, def2/J, RIJCOSX]   # Additional DFT keywords
dft_nprocs: 16                    # Number of processors for DFT calculations
```
